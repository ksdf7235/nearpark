6-1. 새 대화(세션) 시작할 때

이 레포는 skills 디렉터리에 스킬 시스템이 정의되어 있다.

먼저:
- `skills/00_SKILL_SYSTEM.md`
- `skills/project_meta/INSTRUCTIONS.md`

를 참고해서, 앞으로의 모든 작업을 그 규칙에 맞춰 수행해줘.
답변은 항상 PLAN / CODE / NEXT / USED SKILLS 형식을 따라줘.



6-2. 기능을 만들고 싶을 때 예시
skill: supabase_schema

공원 즐겨찾기 기능을 위한 favorites 테이블 설계를 정리해줘.
- 공원 ID, user_id, 메모, created_at 정도 필드가 필요해.
- RLS 정책도 같이 제안해줘.
- `db/schema.sql`에 넣을 수 있게 전체 SQL 코드로 만들어줘.



skill: ui_parks

홈 페이지에 공원 리스트 UI를 만들어줘.
- 공원 이름, 주소를 카드 형태로 보여줘.
- 파일: `src/app/page.tsx`
- 데이터 로딩은 일단 mock 배열로 해도 되고, 나중에 supabase_client 스킬로 바꾸자.


7. 이 구조가 “요약 + 컨텍스트 문제”를 어떻게 막는지

중요 규칙은 모두 md 파일에 고정

대화 요약이 아무리 일어나도 md는 그대로.

스킬마다 책임이 분리

UI 작업 중에 DB 규칙이 헷갈릴 일이 줄어듦.

새 스레드에서도 재사용 가능

레포 + skills 폴더만 있으면, 어디서 시작해도 같은 규칙 적용.

규칙이 바뀌면 먼저 md를 수정

“이제부터 이렇게 해줘” → md에 반영 → LLM 호출
→ 프로젝트 일관성 유지.
